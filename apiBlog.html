What Are APIs?

Application Programming Interface(API) allow applications to connect with other 3rd party applications and utilize their services. For example, Uber developer making use of Google Maps API to allow users to track their cabs.

What is REST?

There are many methods available for creating these APIs. The most popular one which is universally accepted is known as REST(Representational State Transfer). Most web applications make use of REST APIs because of the benefits they offer. Therefore it is important for technical writers to understand how REST APIs work if they are to document web application based APIs successfully.

Note: Basic knowledge of REST is enough. You don't have to know too much about the technicalities involved.

What Do APIs Do? How Do They Look?

They display information that can be used by developers. For example uber developer will make use of Geolocation API to get the 'near by cabs' information and display the same to uber user who is trying to book a cab.

They are basically URLs loaded with information that gets displayed when the URL is accessed. Just like websites where different URLs direct you to different pages, in the case of APIs, different URLs come loaded with different types or sets of information that the developer might search for.

A sample url might look like this: https://api.twitter.com/1.1/. You might have guessed that it belongs to twitter. You might have also guessed that the twitter apis are stored away in subdomain of twitter website.

But a url alone is not sufficient to get started with APIs. You need what is known as an API key.

Note Since APIs deal with sensitive information, they are protected with an API key. The developers looking to subscribe to an API service have to apply for and get the API key first in order to access the service.

APIs - A Case Study

Travel sites like trip advisor serve as good examples of 3rd party API consumers. The information they show on their websites that help you schedule holidays (like available flights, hotels, listing of popular places for sightseeing etc..) are all gathered from several other applications (or websites). These travel websites make money by making the lives of users like you and me (by collating and producing all the information on one screen) and in turn have to pay for the API services that they are subscribe to.

Getting Started - Making a Request, Getting a Response

Let's first explore a free public api (that does not even need an api key) to understand how a API can be accessed and the response it spits out.

All you have to do is click this url: https://randomuser.me/api and the response gets displayed on your browser.

Note: Before accessing the above url, ensure you have this browser extension installed. The extension 'prettyfies' the response.

What is Random User API? What is it reponsible for?

It's just a mock API that serves up random user data, used by designers to quickly wire up a UI without really having to setup a dedicated backend. And it also perfectly suits our purpose of learning the API basics.

How to Make Sense of the Response?

Well, that takes some getting used to. The more you view and analyze, the better. But the good news is you will have a team of developers to help you with this. But the best way to start off is by analyzing easily structured data like the one thrown out by the random user database.

Now if you look at the response closely, you'll realize that it simply gives you the details of a user record like the name, phone number, email, address etc. At a very basic level, this is very much equivalent to a uber customer viewing his/her driver's details. Only difference is, since we don't yet have a frontend, we are being presented with the raw data.

There you go. We just made an API request and got the response back!

Note Everytime you reload the browser, you will get a new random user's details. So take some time and - reload, observe and learn.

Anatomy of an API

An official looks like this: https://api.themoviedb.org/3/movie/550?api_key=6c7d27503ce09c584ac2ee601fd22e1b&language=en

Note Clicking the above url may not give back a response to you. So it's better to get your own API key by visting: https://www.themoviedb.org/documentation/api


Typically, API URLs can be split into 5 sections as follows:

1. Base URL

Base url is that part of the API url that remains the same for all kinds of API calls. It usually refers to the domain which houses all of the application's APIs. The general practice among web applications is to reserve a separate subdomain for hosting their APIs. If you observe the above url, you will realize that it (https://api.themoviedb.org/) is basically a subdomain of (https://www.themoviedb.org/) TMDB web application.

https://api.themoviedb.org/3/

Note If you see a number following the base url, then it refers to API version number. So in our case, it's version 3.

2. API Key

As you can see, the above api requires an api key for it to function. API keys are usually long unique string of characters that cannot be replicated that easily. They ensure only those users (developers) who have subscribed to the API service can access the api, and not anyone else..

api_key=6c7d27503ce09c584ac2ee601fd22e1b

3. Resources

Resources (or resource) refers to the data you are pulling out using the API. It's usually a contextual name given to the data being pulled out from the database, depending on its role within the application (that is offering APIs as a service). In the case of above url, since TMDB provides movie related information, their developers have aptly named the resource as movie.

/movie

4. Path Parameter

To understand this, you have to understand the structure of the URL. Every URL will contain a '?' mark. Values that come after the base url but before the '?' mark are known as path parameters. They are known as path parameters because these values decide the path your API request will take in order to fetch the required data. Path parameters are usually given numerical values that identify a particular resource.

Every resource will have a unique id associated with it. And using these path parameters, you can pass those ids to get the desired resource. In the above url, it's happens to be movie with the id 550 that we are trying to fetch .

/550

5. Query Parameter

They key:pair values that come after the '?' mark are known as query parameters, because they help you 'query' or 'filter the data' being pulled out based on 'some criterion'. After having pulled out the resource, you can decide what you wish to see from the resource using these query parameters.

In the above url, the query parameter used is key:value pair 'language:en' that helps you filter out only English language movies from the list of movies that we got access to from making the API call.

&language=en

Note Multiple query parameters can be used at the same time. Each query parameter has to be separated by '&'

REST API Methods

REST APIs support different types of API operations. But the most popular ones are: POST, GET, PATCH and DELETE.

They basically help you perform CRUD (Create, Read, Update and Delete operations on data) operations via APIs.

POST corresponds to create, GET to read, PATCH to update and DELETE to delete.

Let's see these operations in action.

Available API Methods I

GET Operation


We already made an API call and got the response back by using the below url

https://api.themoviedb.org/3/movie/550?api_key=6c7d27503ce09c584ac2ee601fd22e1b&language=en

What we did was basically a GET operation.

What GET operation does is it queries the database and gets back a specific instance of a resource. Which in our case is the movie with id=550.

GET ALL Operation


The second type of GET operation is known as GET ALL. What this does is it gets back all the instances of a specific resource. Example: let's say the developer wants a list of all the 'trending' movies, then he or she can fetch them all by making use of GET ALL operation as follows:

https://api.themoviedb.org/3/movie/popular?api_key=6c7d27503ce09c584ac2ee601fd22e1b&language=en-US

Note Notice how we no longer pass id but 'popular' path parameter for fetching all the 'trending' movies segregated under 'popular' category.

Note TMDB APIs support only GET operations. They do not allow users to POST data to their database. That means PATCH and DELETE too are not possible. So to explain POST, PATCH and DELETE operations, I'll be making use of my own locally hosted database that serves up mock flight information API data to be consumed by travel website (like tripadvisor) developers.


To reiterate what you have learnt, in the above url:

  • https://api.themoviedb.org/3 is the base url
  • /movie/popular are the path parameters
  • api_key=6c7d27503ce09c584ac2ee601fd22e1b is the api key
  • &language=en-US is the query parameter
